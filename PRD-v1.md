# AI English Learning Desktop Assistant PRD v1

## 1. 文档信息

- 产品名称：AI English Learning Desktop Assistant
- 版本：v1.0 PRD
- 目标平台：Windows 桌面端优先
- 文档目标：定义第一版 MVP 的产品需求、核心流程、技术实现方案和验收标准
- 产品定位：一个嵌入日常电脑使用场景的 AI 英语学习助手，帮助用户从“看到能懂”逐步过渡到“自然能用”

## 2. 背景与问题

传统英语学习工具多以背单词、刷题、词典查询为核心，学习行为和真实使用场景割裂。用户即使背过单词，也经常出现以下问题：

- 在网页、文档、软件界面中遇到英文时，只能临时查词，查完容易忘
- 能看懂单词意思，但写作或表达时想不起来怎么用
- 生词本只记录单词和释义，缺少真实上下文和复用机会
- 复习方式偏机械，难以迁移到实际表达

本产品希望把英语学习嵌入电脑日常使用中，让用户在阅读、查资料、写文档、聊天、看视频等场景里自然接触、理解、收藏、复习并主动使用英文词汇。

## 3. 产品目标

第一版目标不是做一个完整的背单词平台，而是验证一个核心闭环：

用户在真实场景中遇到英文 -> 快速理解 -> 收藏为个人词汇 -> AI 生成复习任务 -> 用户在表达中重新使用 -> 系统记录掌握进度。

### 3.1 核心目标

- 支持用户通过全局快捷键解释选中的英文单词、短语或句子
- 保存生词及其真实上下文，而不是只保存孤立释义
- 使用 AI 生成简短、自然、可复用的解释和例句
- 通过每日小任务推动用户主动使用已收藏词汇
- 支持中文到英文表达改写，帮助用户用上最近学习的词
- 支持在浏览中文网页时，将部分中文词或词组自然替换成英文
- 支持在浏览英文网页时，将部分英文词或词组替换成中文解释
- 鼠标悬停在替换内容上时，展示原文、解释和收藏入口

### 3.2 非目标

第一版暂不追求：

- 完整课程体系
- 大规模词库背诵
- 复杂社交、排行榜、打卡社区
- 自动持续读取用户屏幕内容
- 在所有桌面软件中直接改写屏幕文字
- 多平台同时支持
- 高级语音对话和口语评分

## 4. 目标用户

### 4.1 主要用户

正在使用电脑学习、工作或查资料，并希望在日常使用中提升英语能力的中文用户。

典型用户包括：

- 程序员、设计师、产品经理：经常阅读英文文档、GitHub、技术博客
- 学生和研究人员：经常阅读英文论文、网页资料
- 职场用户：希望提升英文邮件、文档、沟通表达能力
- 自学英语用户：不想机械背单词，希望在真实语境里学习

### 4.2 用户特征

- 有一定英语基础，但主动表达能力不足
- 经常在电脑上遇到英文内容
- 愿意使用 AI 辅助学习
- 对隐私和屏幕读取较敏感
- 更喜欢轻量、即时、低打扰的学习方式

## 5. 核心使用场景

### 5.1 场景一：阅读英文网页或文档时遇到生词

用户在浏览网页、PDF、文档或代码注释时看到不熟悉的英文词。

流程：

1. 用户选中单词、短语或句子
2. 按全局快捷键，例如 Alt + E
3. 应用读取当前选中文本
4. AI 返回解释卡片
5. 用户可以选择收藏

价值：

- 快速理解当前上下文含义
- 不打断当前阅读流程
- 保存真实语境，方便后续复习

### 5.2 场景二：用户想把中文表达得更自然

用户正在写中文笔记、文档或聊天内容，希望知道如何用英文表达。

流程：

1. 用户选中中文句子
2. 按快捷键，例如 Alt + W
3. 应用生成 2 到 3 个英文表达
4. AI 优先尝试使用用户最近学过但尚未掌握的词汇
5. 用户可以复制推荐表达，或收藏其中的词汇用法

价值：

- 帮助用户从被动认识转向主动使用
- 让生词在真实表达里被重新唤醒

### 5.3 场景三：每日轻量复习

用户打开应用或到达设定时间后，应用提供 5 分钟复习任务。

流程：

1. 应用从用户生词库中挑选需要复习的词
2. AI 基于用户的真实上下文生成问题
3. 用户完成释义回忆、句子改写或造句任务
4. AI 反馈表达是否自然
5. 系统更新词汇掌握状态

价值：

- 复习不再只是看中文选英文
- 强调主动输出和自然应用

### 5.4 场景四：日常浏览时自动语境替换

用户在中文网页中阅读内容时，应用自动将少量适合学习的中文词或词组替换成英文表达。例如：

- “这个方案更高效”可以显示为“这个方案更 efficient”
- “我们需要优先处理这个问题”可以显示为“我们需要 prioritize 这个问题”

用户把鼠标放到被替换的英文上时，浮层展示：

- 原始中文
- 英文词或词组
- 当前语境下的含义
- 一个自然例句
- 收藏或忽略按钮

反向场景中，用户在英文网页中阅读内容时，应用可以将少量英文词或词组替换成中文解释。例如：

- “This method is deprecated.” 可以显示为“This method is 已废弃/不推荐使用.”

用户把鼠标放到中文解释上时，浮层展示原始英文、发音、用法和例句。

价值：

- 让英文自然进入中文阅读环境
- 让用户在语境中反复看到同一批目标词
- 通过 hover 保留原文，不破坏理解
- 支持从中文语义到英文表达、从英文阅读到中文理解的双向强化

## 6. 第一版 MVP 范围

第一版建议聚焦 5 个功能模块。

### 6.1 全局快捷键划词解释

用户选中任意软件中的英文文本后，按快捷键触发解释。

功能要求：

- 支持全局快捷键 Alt + E
- 获取当前选中文本
- 支持单词、短语、句子
- 弹出轻量解释卡片
- 支持收藏到生词本
- 支持复制 AI 解释或例句

解释卡片内容：

- 原文
- 中文解释
- 当前上下文含义
- 自然英文例句
- 常见搭配
- 难度标签
- 收藏按钮

### 6.2 生词本

用于保存用户主动收藏的词汇和上下文。

功能要求：

- 展示收藏词列表
- 支持按状态筛选：陌生、被动认识、半熟、主动可用、已掌握
- 展示每个词的原始上下文
- 展示 AI 解释、例句、搭配
- 支持手动修改掌握状态
- 支持删除词条

### 6.3 AI 复习任务

每天根据生词库生成轻量复习任务。

功能要求：

- 默认每天推荐 5 个词
- 每个词生成一个输出型任务
- 支持任务类型：
  - 看英文回忆中文含义
  - 用目标词改写中文句子
  - 用目标词造一句和自己工作或生活相关的话
  - 判断两个例句哪个更自然
- AI 对用户答案给出简短反馈
- 根据结果调整词汇掌握状态

### 6.4 中文转自然英文

用户选中中文句子后，让 AI 给出自然英文表达。

功能要求：

- 支持全局快捷键 Alt + W
- 读取用户选中的中文文本
- 生成 2 到 3 个英文表达
- 标出推荐表达
- 如果适合，优先使用用户最近学习过的词
- 支持一键复制表达

### 6.5 浏览器语境替换

用户在浏览网页时，应用根据页面内容、用户词汇状态和替换强度，自动进行少量双向替换。

第一版范围：

- 优先支持 Chrome / Edge 浏览器插件
- 支持中文网页中的“中文 -> 英文”替换
- 支持英文网页中的“英文 -> 中文”替换
- 支持用户开启、关闭、暂停当前网站替换
- 支持替换强度设置：低、中、高
- 支持 hover 查看原文和解释
- 支持收藏、忽略某个词、恢复页面原文

替换策略：

- 只替换用户正在学习、需要复习或 AI 判断值得学习的词
- 每屏替换数量默认不超过 5 个
- 同一词在同一屏内默认只替换 1 到 2 次
- 不替换输入框、密码框、代码块、按钮、导航、表单、链接文本
- 不破坏原网页结构和可点击行为
- 替换内容需要有轻微视觉标识，避免用户误以为是网页原文

示例：

- 中文原文：这个功能可以提高效率。
- 替换后：这个功能可以 improve efficiency。
- hover 显示：原文“提高效率”，表达“improve efficiency”，例句“This workflow can improve efficiency.”

- 英文原文：The old API is deprecated.
- 替换后：The old API is 已废弃/不推荐使用。
- hover 显示：原文“deprecated”，解释“已废弃；不再推荐使用”，例句“This method is deprecated.”

## 7. 屏幕文本获取方案

第一版采用两种互补方式：桌面场景使用“用户主动触发”，浏览器网页场景使用“用户授权后的页面语境替换”。第一版不做持续屏幕监听。

### 7.1 选中文本 + 剪贴板读取

这是第一版桌面场景的核心方案。

实现流程：

1. 用户在任意应用中选中文本
2. 用户按全局快捷键
3. 应用临时保存当前剪贴板内容
4. 应用模拟 Ctrl + C
5. 读取剪贴板中的选中文本
6. 恢复用户原剪贴板内容
7. 将文本发送给本地处理逻辑和 AI

优点：

- 实现简单
- 覆盖大多数桌面软件
- 用户明确知道什么时候应用在读取内容
- 隐私风险低于持续监听屏幕

限制：

- 无法读取不可复制的图片、视频字幕、游戏画面
- 依赖目标软件是否支持文本复制
- 某些应用可能复制失败

### 7.2 浏览器插件 DOM 读取与替换

这是第一版实现语境替换的核心方案。

实现流程：

1. 用户安装浏览器插件并授权指定网站运行
2. 插件读取网页 DOM 中的可见文本节点
3. 插件跳过输入框、代码块、导航、按钮等不适合替换的区域
4. 插件将候选文本发送给本地应用或 AI Service 进行替换决策
5. 插件将命中的词或词组包装成带 data 属性的 span
6. 页面中显示替换后的文本
7. 用户 hover 时，插件展示原文、解释、例句、收藏和忽略按钮
8. 用户关闭替换或离开页面时，可恢复原始文本

用途：

- 中文网页中的中文词组替换成英文表达
- 英文网页中的英文词组替换成中文解释
- 鼠标悬停查看原文和用法
- 提取当前网页中适合学习的词
- 获取比剪贴板更完整的上下文段落

限制：

- 只覆盖浏览器网页
- 对动态网页需要监听 DOM 变化
- 对复杂页面需要控制替换范围，避免破坏布局和交互

### 7.3 OCR 截屏识别

第一版可作为预研，不进入必做范围。

用途：

- 识别视频字幕
- 识别图片中的英文
- 识别不可复制的软件界面文字

建议后续通过快捷键触发区域截图，而不是自动持续截屏。

## 8. AI 能力设计

### 8.1 单词解释 Prompt 目标

AI 应输出短、准、贴近上下文的解释，而不是长篇词典释义。

输入：

- 用户选中的文本
- 可选上下文
- 用户母语：中文
- 用户词汇历史

输出：

- 中文解释
- 当前语境下的含义
- 常见搭配
- 一个自然例句
- 是否值得收藏
- 难度估计

### 8.2 中文表达改写 Prompt 目标

AI 应帮助用户把中文表达成自然英文，并优先复用用户近期学习的词汇。

输入：

- 中文句子
- 用户最近学习词汇
- 用户词汇掌握状态

输出：

- 2 到 3 个英文表达
- 推荐表达
- 使用了哪些已学词
- 简短说明为什么自然

### 8.3 复习任务 Prompt 目标

AI 应生成短任务，鼓励用户主动输出。

输入：

- 目标词
- 用户原始上下文
- 历史回答情况
- 当前掌握状态

输出：

- 任务类型
- 题目
- 参考答案
- 评分标准
- 简短反馈

### 8.4 语境替换 Prompt 目标

AI 应判断哪些词值得替换、如何替换、替换后是否自然。

输入：

- 页面文本片段
- 页面语言
- 用户正在学习的词汇
- 用户已掌握和已忽略的词汇
- 替换方向：中文到英文，或英文到中文
- 替换强度：低、中、高

输出：

- 候选原文片段
- 替换后的文本
- 原文语言
- 目标语言
- 简短解释
- hover 展示内容
- 是否建议收藏

原则：

- 替换必须自然，不应为了学习强行替换
- 优先替换高频、可迁移、与用户已有词库相关的词
- 不替换会导致句子不通顺或误解的内容
- 避免一段文字中替换过密，影响阅读体验
- 英文替换为中文时，应保留原始英文用于 hover 展示
- 中文替换为英文时，应保留原始中文用于 hover 展示

## 9. 词汇掌握状态

每个词条拥有一个学习状态。

### 9.1 状态定义

- 陌生：用户收藏过，但基本不认识
- 被动认识：看到能理解，但不能主动使用
- 半熟：能大概使用，但不稳定
- 主动可用：能在写作或表达中正确使用
- 已掌握：多次正确使用，近期无需高频复习

### 9.2 状态更新规则

第一版可采用半自动规则：

- 用户收藏新词后，默认状态为陌生
- 用户复习答对含义，状态可提升到被动认识
- 用户能完成改写或造句，状态可提升到半熟
- 用户多次主动使用正确，状态可提升到主动可用
- 连续多轮复习正确且间隔较长，状态可提升到已掌握
- 用户答错或跳过，降低下一次复习间隔

## 12. 技术实现建议

### 12.1 技术栈

推荐第一版：

- 桌面框架：Tauri 或 Electron
- 浏览器插件：Chrome / Edge Extension Manifest V3
- 前端：React + TypeScript
- 本地数据库：SQLite
- AI 接口：OpenAI API 或兼容接口
- 快捷键：桌面框架提供的 global shortcut 能力
- 剪贴板：系统剪贴板 API

### 12.2 推荐架构

模块：

- Desktop Shell：托盘、窗口、全局快捷键
- Clipboard Service：读取选中文本、恢复剪贴板
- AI Service：统一封装解释、改写、复习任务
- Browser Extension Service：网页文本读取、替换、hover 浮层
- Replacement Service：替换候选选择、替换强度控制、忽略规则
- Vocabulary Service：词条增删改查
- Review Service：复习计划和状态更新
- UI Layer：浮窗、生词本、复习页、设置页

### 12.3 隐私策略

第一版必须遵守：

- 默认不持续读取屏幕
- 默认不自动上传用户屏幕内容
- 只有用户按快捷键触发时才读取选中文本
- 浏览器插件仅在用户授权的网站运行
- 语境替换必须提供明显的开启、暂停和恢复原文入口
- 浏览器插件发送页面文本前，应在设置中说明发送范围和用途
- 默认只处理正文可见文本，不处理输入框、表单、密码框和隐藏内容
- AI 请求前可在设置中显示发送内容范围
- 本地生词库默认保存在用户电脑
- 可提供“敏感应用黑名单”，在指定应用中禁用快捷键
- 可提供“敏感网站黑名单”，在指定网站中禁用语境替换

## 15. 成功指标

MVP 阶段重点看行为闭环是否成立。

核心指标：

- 每日解释次数
- 收藏率：解释后收藏的比例
- 复习完成率
- 主动输出任务完成率
- 从“陌生”提升到“半熟”的词汇数量
- 中文改写中复用已学词的次数
- 网页语境替换曝光次数
- hover 查看率
- 替换后收藏率
- 用户主动暂停替换的比例
- 用户 7 日留存

定性指标：

- 用户是否觉得学习没有明显打断工作
- 用户是否开始在真实表达中使用学过的词
- 用户是否愿意把它作为常驻桌面工具

## 16. 风险与应对

### 16.1 剪贴板读取不稳定

风险：

- 某些应用不支持复制
- 模拟 Ctrl + C 可能失败

应对：

- 失败时提示用户手动复制后重试
- 浏览器网页场景通过插件补充剪贴板方案
- 后续增加 OCR 识别不可复制的图片、字幕和软件界面文字

### 16.2 AI 输出过长或不稳定

风险：

- 解释太长，打断用户
- 例句不自然或难度过高

应对：

- 使用结构化 Prompt
- 限制输出长度
- 为解释、改写、复习分别设计模板

### 16.3 隐私担忧

风险：

- 用户担心应用读取屏幕或上传隐私内容

应对：

- 默认仅快捷键触发
- 设置页明确展示隐私策略
- 提供敏感应用黑名单
- 提供敏感网站黑名单
- 保留本地数据导出和删除能力

### 16.4 学习闭环不够强

风险：

- 用户只查词，不复习，也不输出

应对：

- 解释后提供“用一下这个词”的轻任务
- 每日复习强调主动表达
- 中文改写时主动唤醒最近学过的词

### 16.5 语境替换干扰阅读

风险：

- 替换太多会影响阅读效率
- 替换不自然会让用户出戏
- 中英文混排可能造成理解负担

应对：

- 默认使用低强度替换
- 每屏限制替换数量
- 提供一键暂停和恢复原文
- 允许用户忽略某个词或整个网站
- 替换内容保留轻微视觉标识

### 16.6 网页结构被破坏

风险：

- DOM 替换可能破坏链接、按钮、代码块或动态页面

应对：

- 跳过交互元素、表单元素、代码块、导航区域
- 使用 span 包装文本节点，不修改原有元素结构
- 使用 MutationObserver 处理动态内容
- 每次替换保留原始文本，支持恢复

## 17. 第一版开发建议

建议按以下顺序实现：

1. 搭建桌面应用和本地数据库
2. 实现全局快捷键和剪贴板读取
3. 接入 AI 解释接口
4. 完成解释浮窗和收藏
5. 完成生词本
6. 完成浏览器插件基础通信
7. 完成网页语境替换
8. 完成每日复习
9. 完成中文转英文改写
10. 补充隐私设置和错误处理

第一版应该优先把“遇到词 -> 理解 -> 收藏 -> 复习 -> 用出来”的闭环跑通，而不是过早扩展大量学习内容。
